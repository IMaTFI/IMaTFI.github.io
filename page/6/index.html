Learning Relaxed Belady for Content Distribution Network Caching学习笔记

Posted on 2022-07-23 Edited on 2022-07-28 In lab

参考资料：

  • Learning Relaxed Belady for Content Distribution Network Caching

Abstract

CDN：Content Distribution Network
LRB：learning Relaxed Belady
WAN：Wide Area Network

1 Introduction

2022年，CDN在网络传输中的占比预计高达72%

CDN的miss会带来额外成本，为此要设法降低byte miss ratio。

目前CDN使用的最先进的cache算法是启发式的LRU算法，该算法在某些情景下表现较好，但在其他情境下表现较差。

尽管关于cache的研究已经进行了50年，但在byte miss ratio方面，online算法和offline算法之间仍有很大的差距。

Belady算法旨在淘汰距离下一访问最远的object，而Relaxed Belady则随机淘汰距离下一次访问超过一定界限的object。本文中，作者使用机器学习的方法对Relaxed Belady进行了近似。

何为heuristics？

Belady boundary：被Belady算法淘汰的object中，距离下一次访问最近的时间。将Belady boundary作为Relaxed Belady中的界，可以很好的近似Belady算法。

byte miss ratio是一种end-to-end的评价标准，这意味着对于一种配置，要经过几个小时的长期跟踪才能对其进行评判。

good decision ratio：淘汰的object中，距离下一次访问超过界的比例。作者以good decision ratio为最终评价标准，对机器学习模型进行训练。

2 Background and Motivation

2.1 The Challenge of Reducing WAN Traffic

寻找方法去降低CDN的byte miss ratio

CDN cache的容量很紧张，object之间的竞争很激烈

大约有75%的object被访问后的两天内都不会再被访问，称为one-hit-wonder。让这样的object进入cache无疑会造成空间浪费。CDN的cache算法提升空间很大。

CDN的workloads随地理位置、时间而显著变化，所以需要能随时间和访问模式而变化的cache算法。

2.2 Opportunity and Requirements


尽管CDN cache的空间很紧张，但CPU算力很充足，可以额外容纳一些复杂的cache算法。

三个限制条件：

  1. 存储空间只有几GB
  2. 没有TPU和GPU
  3. 每秒处理数万个请求

3 Approximating Belady Algorithm

3.1 Relaxed Belady Algorithm

机器学习算法很难近似Belady算法。首先，计算每一个object的下一次访问时间的成本过高；其次，精确预测object的下一次访问时间很难。

Relaxed Belady algorithm根据到下一次访问的距离是否超过一定的界限，将cache中的object划分两部分A、B。若A非空，则从A中随机淘汰；若A为空，则对B执行Belady算法。

Relaxed Belady algorithm有两个重要性质：

  1. 当界限设得很大时，算法的byte miss ratio和Belady算法很接近。
  2. 每次淘汰时有很多选择。

基于上述两条性质，ML算法的时间消耗和空间消耗能显著减少，因为ML算法只需要在objects的一个小子集中进行选择即可，只要这个集合中有符合淘汰条件的object。ML算法的精度也不需要很高，实质上只需要判断object的下一次访问时间是否越界即可。

找到合适的界限很重要。

3.2 Belady Boundary

Belady boundary：被Belady算法淘汰的object中，到下一次访问距离最近的时间。

Belady boundary的计算需要未来的知识，实际使用中，只需根据训练集设定界限即可。

由表2可以看出，relaxed Belady相比于Belady的差距是可接受的。

3.3 Good Decision Ratio

byte miss ratio可以从总体上评价决策的好坏，但不能知道哪些是好决策、哪些是坏决策。

good decision ratio：淘汰的object中，到下一次访问的距离超过界的比例。

4 Design of Learning Relaxed Belady Cache

ML算法中的4个关键方面：

  1. 决定多少来自过去的信息是可用的。
  2. 如何根据过去的信息动态产生由标签的训练集。
  3. 选择好的ML模型，进行特征选择、目标预测、损失计算、淘汰决策。
  4. 如何选择候选者。

ML算法的目的是提高good decision ratio

4.1 Past information

LRB只保存最近一次request出现在sliding memory window内的object的信息，并用sliding memory window（后称滑动窗口）内的信息进行训练和预测。

这并不意味着没在滑动窗口内的object没有特征值。要注意past information和training data的区别。但不在窗口内的object的信息比较少，只有静态信息和距离上一次访问的距离。

LRB通过验证集选择合适的滑动窗口大小。对于小cache，验证集可以达到最优化滑动窗口的效果，使其能最大化good decision ratio；对于大cache，使用最小二乘法拟合出的直线来选取窗口大小。

4.2 Training Data

构建训练数据是有挑战性的：特征随时间不断变化，且label直到下一次request出现前都不能确定。因此，LRU先产生没有标签的数据。

LRB每隔一段时间便从滑动窗口中选取一些object，记录下这些object的特征。作者之所以不根据request选取，或者只从cache中的object中选取，是为了避免对popular object的偏好。

LRB用两种方法标记数据：

  1. 等到该object下一次被request。这种方法存在明显的缺陷，比如这种方法永远不能得到只被request一次的object的标签，而这样的object恰恰是需要终点学习的。
  2. 当该object距离下一次被request的时间已经超过了Belady boundary，就将其标记。LRU使用这种方法标记滑动窗口以外的object。

4.3 ML Architecture

4.3.1 Features：Deltas，EDCs，and Static

Deltas：代表上一次request到现在的距离，前第n次request到前第n-1次request的距离。如果该object只被request了n次，则。LRU实质上就是根据来淘汰object。

EDCs：exponentially decayed counters。每个EDC均初始化为0。每当一个request到来时，更新。只有在该object的request到来时才更新，而Deltas则是每当request到来时就更新。观察上述公式可知，越小，衰减越快。所以如果有一个object，在过去十分流行，而最近不流行，则其相对较小，相对较大。

Static features：object的大小、类型等。这些特征容易得到且内存消耗很小。

作者为object设置了32个Deltas和10个EDCs

4.3.2 Model：Gradient Boosting Machines

4.3.3 Prediction Target：log（time-to-next-request）

LRB的预测值为到下一次request的距离。LRB实际上只对sliding memory window内的object进行预测，其余object的预测值直接设置为window大小的2倍。

为啥不用二分类模型呢？感觉二分类模型似乎应该更好？为什么二分类反而表现更差？

4.3.4 Loss Function：L2

mean square error

4.3.5 Training data Size：128K

每当训练数据达到128K时，LRB就训练一个新的模型，然后清空训练数据。

4.4 Eviction Candidate Selection

LRB随机从cache中的objects选择64个对其进行预测，淘汰其中预测值最大者。这个过程只需要30微妙。

这可能就是回归模型比二分类模型更优越的原因，回归模型能淘汰越界object中更值得淘汰的那一个。

4.5 Putting All Together


如上图所示，LRB根据长度和Belady boundary相似的滑动窗口内的objects进行学习。这些object的特征被保存在一个紧凑的数据结构中，当新的请求到来时，更新object的特征，同时移动滑动窗口。LRB持续地从上述的数据结构中对数据进行采样，当训练数据集达到128K时，LRB开始训练GBM模型，同时清空训练数据集。当训练数据集再次达到128K时，LRB便开始训练新模型。如果当前的request触发了cache miss，LRB随机从cache中选取64个object并使用GBM模型对其进行预测，淘汰距离下一次访问的预测值的最大者。

Caching with Delayed Hits学习笔记

    Abstract

    delayed hit的存在会造成缓存系统的实际延迟比预测延迟要高。当delayed hit存在时，传统的cache优化策略并不能使latency最小。新的offline优化策略BELATEDLY能有效解决这个问题。并根据BELATEDLY设计了online算法MAD，有效降低了latency。

1 Introduction

delayed hit：请求1需要取出一个cache中没有的object，在处理的miss的过程中，请求2到来了，且同样需要取出和上一个请求相同的object。显然，请求2的处理时长比普通命中的处理时长要高，但系统仍认为其命中，这就是delayed hit。

offline-optimal假设能够完全预测未来，所以能得出实际cache的命中率上限。offline-optimal能指导实际的cache。本文中讨论的也是一种offline算法。

传统Belady’s algorithm能最大化命中率，但实际上并不能最小化延迟。

MAD：Minimum Aggregate Delay

Z：object fetch time / mean request inter-arrival time

Z越大，BELATEDLY相比传统方法越优越。

2 The Problem with Delayed Hits


图1形象地展示了delayed hit的概念。

2.1 Classic Caching Algorithm

caching algorithm：决定哪些对象要载入到缓存中，哪些对象要淘汰。

在delayed hit存在的情况下，传统模型会严重低估cache的延迟。

2.2 Belady is Not Latency-Minimal

在无delayed hits、对象大小一致、访问backing store延迟一致的情况下，Belady’s algorithm 被证明是最优算法。算法思想为：淘汰距离下一次访问最远的object。

由图3给出了一个delayed hit存在的情况下，Belady不是最优解的情况。

2.3 Delayed Hits and Practical Algorithm


当delayed hits不常发生时，传统预期延迟和真实延迟差距很小。

当Z>100时开始出现差异，当Z>1000时差距逐渐明显

当Z增大时，预测和实际的差距也在增大。

“红叉”表示：在两个算法之间，只根据hit ratio选择，平均延迟表现更糟的情况（即选错的情况）。可以看出，有超过1/3的情况都选错了。

delayed hits 的存在可能会对人们的决策选择产生误导。

2.4 Minimizing Latency is challenging

有例子表明，随着Z的增长，最优策略可能会产生很大的变化。

在考虑delayed hits时，一个真正可以降低latency的算法可能会降低命中率，或者提高缺失率。

RobinHood: Tail Latency Aware Caching-Dynamic Reallocation from Cache-Rich to Cache-Poor学习笔记

Abstract

造成tail latency原因：单个用户请求可能会查询多个后端服务器，而不同的服务器的查询延迟差别高达2个数量级。

RobinHood通过将缓存资源从缓存丰富的服务器向缓存缺乏的服务器调度，来改善tail latency问题。

1 Introduction

P99：前99%快的延迟时间中的最长延迟。

用户的request被处理成多个query，交由多个backend处理，当所有query处理完毕后，用户才收到结果。所有query并行处理，tail latency取决于最慢的query。

tail latency的来源是动态的

由图2可知，不同backend在不同时间都出现了高延迟。

一些现有方法致力于平衡server之间的负载，从而平衡backend之间的tail latency。但受限于结构约束，某些query只能由特定的backend处理，所以在不同backend之间平衡负载是不可能的。
另一种方法是将服务器分配给高延迟的backend。由于backend的延迟情况变化很快，所以这种方法需要快速扩展backend，但这是很困难的。

RobinHood通过动态分配cache空间来降低整体的tail latency，但可能会造成某些backend的tail latency的提高。由于许多cache系统支持空间分配，所以使用RobinHood是很方便的。

RobinHood不是传统的cache系统。传统cache系统通过保证popular object的命中，减少query的平均延迟。而request的尾延迟往往是由miss造成的。

2 Background and Challenges

2.1 The goal of RobinHood

实现RobinHood的关键点在于找出造成高P99延迟的backend，即cache-poor backend。然后将cache资源向poor处转移。RobinHood会牺牲一些backend的延迟，从而获取闲置的cache空间。
获得额外的cache空间后，cache-poor backend对应的cache的命中率提高，访问这些backend的query就会变少。

换句话说，如某个backend的表现不好，就尽可能减少其工作量。

另外，减少query还有助于减少backend之间的资源竞争，从而显著减少P99。

2.2 Challenges of caching for tail latency

2.2.1 Time-varying latency imbalance


不同后端服务器延迟不同，相同后端服务器不同时间延迟不同。某些backend很慢，因为它不止为OneRF服务。

这可能也解释了为什么其他方法不能有效改善tail latency的问题。

传统缓存系统假设延迟平衡。而当不同后端延迟不同时，cache miss的代价是不一样的。

2.2.2 延迟与特定查询或查询率无关

高延迟与特定的query无关、与query的流行程度无关，它更像是反映了backend的一种状态。

图4说明了latency与query的流行度无关。

此外，latency也和查询率无关。某个backend的查询率低，并不意味着其计算资源丰富，因为它很可能正在被其他业务所占用。

是否存在某种方法，能统计backend上所有不同来源的query？这样或许就能揭示查询率和latency之间的关系。

传统方式（多个后端服务器共享一个cache空间，共用一个淘汰机制）和cache分割方式都倾向于将大的cache空间分配给查询次数较多的后端处理器。然而根据上文可知，查询率的高低与延迟无关，所以好的cache应着重考虑查询延迟。

2.2.3 Latency depends on request structure，which varies greatly

request structure：request到query的映射。
batch size：单个backend处理的query数量。（对一个request而言，吗？）
fanout：单个request查询的backend的数量。

对于一个给定的backend，作者研究了这个backend上batch size的平均值和涉及到这个backend的request的fanout的平均值。

为什么要研究这些？
作者是怎么证明tail latency和request structure有关的？


由表1可知，不同backend之间的batch size和fanout差别很大。

传统方法下，即使以P99作为优化标准对存储系统进行优化，效果也不明显。如果一个backend只处理很少量的query，即使其延迟很高，也不会对P99产生多大的影响，但一个同时考虑查询率和延迟的系统可能会为这个backend分配大量的cache空间，然而这并不能提升P99。

RobinHood为什么可以避免这种现象？是因为集合S的选取吗？如果其他算法也只关注集合S，是不是也能避免这个问题呢？
似乎不是集合S的原因，因为对于RobinHood而言，即使你的延迟再高，最多也只能使你的RBC+1！

3 The RobinHood Caching system

3.1 The basic RobinHood algorithm

RobinHood回收每个backend的1%的cache space，称为“税金”。根据哪些backend缺乏cache space来将“税金”重新分配。
1%是怎么来的？是否可以通过一种机器学习算法，计算出更好的比例？不过好像不太重要。。。

RobinHood维持一个长度为秒的窗口，在此期间对每个request的latency进行监控。RobinHood监测P98.5到P99.5的requests，称为集合S，并找出每个request中响应最慢的backend，然后统计每个backend响应最慢的次数，称为request blocking count（RBC）。RobinHood依据RBC进行“税金”分配。

为什么选择RBC作为评价标准呢？如上文所说，只关注backend的latency是不够的，因为慢并不意味着最慢，backend的延迟会因为其batch size和fanout而被隐藏或放大。举例来说，如果一个backend的latency较高，但涉及该backend的request的fanout很大，则该backend成为最慢者的概率相对较低。一个backend的RBC很高，意味着其造成了高延迟，还意味着改善其延迟将有效地降低整体延迟。

RBC好就好在，它不仅考虑了延迟，还将request structure也考虑了进来。

集合S为什么没有直接设置为>P99的request呢？因为>P99的request中有“outliers”，这些“outliers”的latency即使经过显著优化，也依然远大于P99，这对于优化P99是没有意义的。

选择P99~P99.5不好吗?

3.2 refining the RobinHood algorithm

两个改进：

  1. basic RobinHood假设每个backend都能迅速填满新增的cache space，而实际情况中，某些backend的hit ratio已经很高，导致额外的cache space不能很好地利用。假如某些backend的已分配空间与已使用空间的差值占已使用空间的比例超过30%，RobinHood会忽视这些backend的RBC，转去优化其他的backend。
    换句话说，这种情况下RobinHood失效了。可以对这种情况进行进一步研究，如果这种情况经常出现的话，则可以考虑通过其他手段去解决这些backend的延迟问题。
  1. basic RobinHood假设所有backend都是一个大cache的一部分，而实际的系统中可能有多个application server（这些server共用backend）。所以RobinHood在每个server上都安排了一个本地的controller。假设，不论server的cache的初始状态如何，只要RBC数据相同，RobinHood都能在30min内达到几乎相同的分配策略。

3.3 RobinHood architecture

使用RobinHood需要在现有系统上增加两个额外部分。首先是一个轻量的cache controller，以实现在server上的缓存调度。controller的输入是RBC。然后是一个集中式的RBC server，用于统计每个backend的RBC。

RobinHood可以很快地从突发错误中恢复，且成本低、不会造成额外延迟。RobinHood controller和RBC server不在request和query的关键路径上，不会造成额外成本。

4 System Implementation and Challenges

4.1 Implementation and testbed

RobinHood controller是一个轻量的python进程，从RBC server接收RBC数据，计算cache size。RBC server用Go语言实现。在作者的测试中，所有cache都具有动态分配的功能，每个application都有32GB的cache容量。

作者使用了不同的backend系统和一个能持续发送request的generator，每个request平均“分裂”成50个query。一个query首先在本地cache中查找，miss后进入相应的backend。在作者的实验中，每秒有20万个query（峰值为50万）

4.2 Implementation challenges

作者遇到最大的挑战是如何扩展他们的系统，是其能横跨20个不同的backend系统，处理20万-50万每秒的query数。作者放弃了碎片化的cache结构，因为这意味着每个query都要查询所有cache，造成过高的时间成本

既然如此，sharded cache的优点在哪里呢？部署方便？使用和规模较大的工作场景？

  1. 非凸的miss curves：虽然RobinHood的运行不依赖miss，但miss率的激增可能会导致RobinHood效果下降。

这篇论文解在维持低尾延迟方面解决了两个问题。第一个问题是系统资源的分配问题，RobinHood为每个backend引入了RBC的概念，从而识别出哪些backend需要额外的资源。第二个问题是解决有状态后端系统之间的延迟不平衡问题，这些系统无法直接扩展以利用额外的资源。RobinHood利用了多层系统中现有的缓存层，将缓存空间差异分配给各个后端，而不是直接扩展它们。

当浏览器加载某些特定的图片时，一段javascript程序就会向Scribe发送报告。

当有HTTP回复发送回client时，Edge Cache便向Scribe发送报告。即使request在Edge Cache未命中，Edge返回的信息中也会包含来自下游的一些信息，所以Edge Cache发送的报告同时包括Edge Cache和Origin Cache的命中/缺失信息。

当某个请求在backend上完成时，Origin host便向Scribe发送报告。

我们对于信息的采集，让我们实现了如下目标：

  2. 探究request沿cache层次结构传递的路径

但我们的采集方法也带来了一些挑战：

  1. 判断browser cache的命中是十分困难的。为此，我们通过对比相同URL在浏览器和Edge上的数量，来判断命中。

3.3 Sampling Bias

有两种采样策略：

  1. 随机对requests进行采样。
  2. 对图片的id进行随机采样。

作者选择后者，出于以下两个原因：

  1. 避免了对流行图片的偏好
  2. 可以通过id号很方便观察图片在不同layer中的情况。

但这种方法亦有缺陷。最终实验证明，采样下的cache表现与实际cache表现类似，所以可以认为没有偏差。

为什么不能对requests进行采样呢？虽然request中，popular图片占比会更多，但这不是恰好符合现实吗？不是更有助于对实际情况进行分析吗?

为什么根据id进行采样，不同popular level的图片的比例仍然会有不同？workload该如何理解？每张图片（包括其transformation）在workload中会有重复吗？

一种可能的解释：workload和request本质上是一样的，但前者是与时间无关，后者与时间有关。作者之所以认为对request采样会对“暂时流行”的图片有偏好，是因为采样的那段时间很可能是其唯一流行的时间。

4 Workload Characteristics


观察“Photos w/o size”，我们发现各个层级的图片数量接近，大约都为1.3M。相邻两个层级的差值，代表了在前一层处理的图片数量。所以对于1.3M不流行的图片，往往在每层cache中都miss了。

观察“Photos w/ size”，我们发现Origin cache和Backend之间的差值最大，考虑到Origin cache层具有裁剪图片的功能，说明前几层的miss中，有相当一部分都是由于图片大小不符。

5 Geographic Traffic Distribution

5.1 Client To Edge Cache Traffic


图5展示了每个城市的用户发送的request最终被不同Edge Cache处理的比例。注意到，每座城市的request均被所有的Edge Cache服务，即使有些Edge Cache距离当前城市很远，有着很高的延迟。并且，城市的requests中的大多数往往不是被距离最近的Edge Cache处理的。

上述现象是由基于总延迟、Edge Cache容量、ISP成本的路由政策造成的。当收到request时，DNS服务器会综合考虑延迟、“交通”状况、“交通”花费来选择最佳的Edge Cache。

由于Facebook的Edge Cache分配政策，同一用户可能在不同的时间段内被不同的Edge Cache服务，这导致了Cache命中率的下降。

5.2 Edge Cache to Origin Cache Traffic

Facebook的照片服务集中在4个数据中心，每个数据中心都有Origin Cache，所有的Origin Cache均被视作一个整体。

图6描述了每个Edge Cache向不同数据中心发出的request所占的比例。由图可知，各个城市的比例分布几乎相同，再次证明了这4个Origin Cache的整体性。

5.3 Cross-Region Traffic at Backend

有两种情况会导致request不能在当前backend处理：

  1. 由于Facebook经常对Backend的数据进行迁移，所以偶尔会发生无法在当前backend找到目标的情况。
  2. 当前backend出于脱机或过载而无法响应请求。

以上两种情况出现的概率一般不超过0.5%。

6 Potential Improvements

6.1 Browser Cache


如图8所示，越活跃的用户，其browser cache的命中率越高。总的命中率为65.5%.

作者假设browser cache拥有无限的容量，从而将cold miss和capacity miss（由于容量不足而淘汰object造成的miss）区分开来，其结果给出了提升cache容量和改进淘汰算法带来的提升的上界。由上图可知，不活跃用户不会因为cache无限的容量而获益（由此可见，这些用户的miss大多为cold miss）。

作者同样假设resize的工作由用户完成，这种情况下，不活跃用户的提升程度较大。

我认为，browser cache的提升空间并不大，且提升cache space或resize都会给用户带来体验上的损失。

作者着重研究了9个常用的Edge Cache，“All”代表所有Edge Cache的总hit ratio，“Coord”代表一个假设的、由这9个Edge Cache合体而成的Edge Cache的hit ratio。
Edge Cache的提升空间就很大，可以重点研究。

图10（a）、（b）中的灰线即原命中率，横轴上的“x”即原容量。这里，作者解释了“sheltering”和“bandwidth reduction”的区别：前者致力于减少向下一层的request数，即提高object-hit ratio；后者致力于提高byte-hit ratio。由图10可知，S4LRU是不错的淘汰算法，但提升空间仍然很大。

作者再次讨论了“合体版”Edge Cache的情况，出于以下两点考虑：

  1. 现有结构下，同一照片可能在不同的Edge Cache中均有存储，而“合体”后只需存储一张即可，从而留出更的空间存储其他图片。
  2. “合体版”Edge Cache可以避免由于用户在不同Edge Cache之间的切换而造成的cold miss。
