9.1.2.2 连接操作的实现

以 SELECT * FROM Student, SC WHERE Student.Sno=SC.Sno为例

循环嵌套算法

对外层循环（Student表）中的每个元组，检查内层循环（SC表）中的每个元组，检查这两个元组是否满足连接条件，如果满足则串接后作为结果输出，知道外层循环的所有元组处理完为止。

排序-合并算法

等值连接的常用算法，尤其适合参与连接的表已经排好序的情况。

一般步骤：

  • 对参与连接且没有排序的表按照连接属性进行排序
  • 取Student表中的第一个Sno，以此扫描SC表中具有相同Sno的元组，把它们连接起来。当扫描到Sno不相同的一个SC元组时，返回扫描Student表中的下一个元组，直到Student表全部扫描完。

索引连接算法

一般步骤：

  • 在SC表上建立Sno索引
  • 对Student中的没一个元组，由Sno值通过SC索引查找相应的SC元组
  • 把这些SC元组和Student元组连接起来

hash join 算法

将连接属性作为hash码，用同一个hash函数将Student表和SC表中的元组散列到hash表中。

一般步骤：

  • 划分阶段：对元组较少的表（如Student表）进行一遍处理，把它的元组按hash函数散列到hash表的桶中
  • 试探阶段：对另一个表进行一遍处理，把它的元组按hash函数散列，找到对应的桶，把SC元组与桶中相匹配的Student元组连接起来

数据库——绪论

Posted on 2023-02-21

1.1 数据库系统概述（P3）

1.1.1 数据库的四个基本概念（P3）

  • 数据：描述事务的符号记录
  • 数据库（DB）：长期存储在计算机内、有组织的、可共享的大量数据的集合。具有永久存储、有组织和可共享三个基本特点。
  • 数据库管理系统（DBMS）：计算机基础软件，主要具有如下功能：
    • 数据定义功能
    • 数据组织、存储和管理
    • 数据操纵功能
    • 数据库的事务管理和运行管理
    • 数据库的建立和维护
  • 数据库系统（DBS）：由数据库、DBMS、应用程序、DBA组成的存储、管理、处理和维护数据的系统。

1.1.2 数据管理技术的产生和发展（P6）

  • 人工管理阶段：
    • 数据不保存
    • 应用程序管理数据
    • 数据不共享
    • 数据不具有独立性
  • 文件系统阶段：
    • 数据可以长期保存
    • 由文件系统管理数据
    • 数据共享性差、冗余度大、独立性差
  • 数据库系统阶段

1.1.3 数据库系统的特点（P9）

  • 数据结构化
  • 数据的共享性高、冗余度低、易扩充
  • 数据独立性高：
    • 物理独立性：用户的应用程序与数据库中数据的物理存储是相互独立的。
    • 逻辑独立性：用户的应用程序与数据库的逻辑结构是相互独立的。
  • 数据由数据库管理系统统一管理和控制：安全性保护、完整性检查、并发控制、数据恢复

1.2 数据模型（P14）

数据模型：对现实世界数据特征的抽象，是数据库系统的核心和基础

数据模型三要素：数据结构、数据操作、数据约束（完整性约束）

1.2.1 两类数据模型（P15）

  • 概念模型：按用户的观点来对数据和信息进行建模
  • 逻辑模型：按计算机系统的观点对数据进行建模，主要用于数据库管理系统的实现。如：层次模型、关系模型、面向对象模型等。
  • 物理模型：数据在系统内部的表示方式和存取方法。

    概念模型为一类，逻辑模型和物理模型为一类。

1.2.2 概念模型（P16）

信息世界的主要概念：

  • 实体：客观存在并可相互区别的事物
  • 属性：实体具有的某一特性
  • 码：唯一标识实体的属性集

  • 实体型：实体名及其属性名集合

    例如，学生（学号，姓名，性别）

  • 联系：实体内部和实体之间的联系

  • E-R模型：实体-联系方法

1.2.7 关系模型（P25）

优点：

  • 建立在严格的数学概念的基础上
  • 关系模型概念单一，所以其数据结构简单、清晰，用户易懂易用
  • 关系模型的存取路径对用户透明，从而具有较高的数据独立性和安全保密性，简化了程序员的工作和数据库开发建立的工作

缺点：

  • 查询效率较低

1.3 数据库系统的结构（P27）

1.3.2 数据库系统的三级模式结构（P28）

  • 模式：又称逻辑模式，是数据库中全体数据的逻辑结构和特征的描述，是所有用户的公共数据视图。

    数据记录由哪些数据项构成；数据项的名称、类型；数据之间的联系

  • 外模式：又称子模式或用户模式，是数据库用户能看见和使用的局部数据的逻辑结构和特征的描述，是数据库用户的数据视图，是与某一应用有关的数据的逻辑表示。

    外模式通常是模式的子集

  • 内模式：又称存储模式，是数据物理结构和存储方式的藐视，是数据在数据库内部的组织方式

    记录的存储方式是堆存储还是按某个值升序存储；使用B+树索引还是hash索引

1.3.3 数据库的二级映像功能与数据独立性（P29）

  • 外模式 / 模式映像：当模式改变时（如增加新的关系、新的属性、改变属性的数据类型），由数据库管理员对各个外模式 / 模式映像做相应的改变，可以使外模式保持不变，从而使应用程序可以不用修改，保证了数据的逻辑独立性。
  • 模式 / 内模式映像：当数据库的存储结构改变时（选用了另一种存储结构），由数据库管理员对模式 / 内模式映像作相应改变，可以是的模式保持不变，从而应用程序页不必改变，保证了数据的物理独立性。

数据结构化、数据高度共享、数据独立性高、安全性、完整性、恢复能力、数据一致性。

p262 8-1 什么是“设备独立性”？引入这一概念有什么好处？
答：设备独立性指，用户在程序中使用的设备与实际使用的设备无关，也就是在用户程序中仅使用逻辑设备名。引入设备独立性的好处：方便用户使用、改善设备利用率、提高系统的可扩展性和可适应性。

p262 8-3 什么是设备控制块？它主要应包含什么内容？简述其作用。
答：系统为每一台设备都配置了一个用来记录设备的硬件特性、连接和使用情况的一组数据，称为设备控制块。主要内容：设备名、设备属性、命令转换表（命令名、例程地址）。

p262 8-7 对I/O设备分配的一般策略是什么？
答：独享分配、共享分配、虚拟分配。

p262 8-8 什么是独占设备？对独占设备应如何分配？
答：独占设备即让一个作业在整个运行期间独占使用的设备。独占设备通常采用独享分配，在一个作业执行前，将它所要使用的设备分配给它；当它结束时，将分配给它的这类设备回收。

p262 8-9 什么是共享设备？对共享设备应如何分配？
答：共享设备即由多个作业、进程共同使用的设备。共享设备一般采用共享分配，在进程提出资源申请时，将所要使用的设备分配给它；进程使用完毕后，立即将设备收回。

p262 8-10 什么是虚拟设备技术？什么是虚拟设备？如何进行虚拟分配？
答：虚拟技术是在一类物理设备上模拟另一类物理设备的技术，是将独占设备转化为共享设备的技术。通常把用来代替独占性设备的那部分外存空间称为虚拟设备。虚拟分配的方法为：当进程需要与独占设备交换信息时，系统将分配磁盘空间，并建立相应的数据结构。

4.1 连续分配方式

要求：每个文件在磁盘上占有一组连续的块。

优点：支持随机访问；顺序读/写速度最快（磁头移动距离短）。
缺点：不利于文件扩展；物理空间利用效率低，产生难以处理的磁盘碎片。

4.2 链接分配方式

隐式链接：

优点：磁盘利用率高，没有磁盘碎片；方便文件扩展。
缺点：只支持顺序访问，不支持随机访问。

显式链接：

优点：由于FAT表常驻内存，当我们想要访问某个逻辑块时，可以通过查表的方式找到其对应的物理块，所以显示链接是支持随机访问的；不会产生磁盘碎片；方便文件扩展。
缺点：FAT表需要占用一定的存储空间。

4.3 索引分配方式

优点：支持随机访问；方便文件扩展。
缺点：索引表占用存储空间。

数据库——数据库安全性

Posted on 2023-02-08

问题提出：

  • 数据库的一大特点是数据可以共享，但不是无条件的共享。
  • 数据库的数据共享在DBMS的严格控制之下，只允许有合法使用权限的用户访问相应的数据。

4.2 数据库安全性控制（P137）

4.2.1 用户身份鉴别（P138）

用户身份鉴别是系统提供的最外层安全保护措施。

基本方法：

  • 系统提供一定的方式让用户标识自己的名字或身份；
  • 系统内部记录着所有合法用户的标识；
  • 每次用户要求进入系统时，由系统核对用户提供的身份标识；
  • 通过鉴定后才提供机器使用权。
  • 用户标识和鉴定可以重复多次

4.2.2 存取控制（P140）

存取控制机制包括定义存取权限和检查存取权限两部分，二者共同构成DBMS的存取控制子系统。

  • 定义用户权限：在数据库系统中，为了保证用户只能访问他有权存取的数据，必须预先对每个用户定义存取权限。
  • 合法权限检查：对于通过鉴定获得上机权的用户（即合法用户），系统根据他的存取权限定义对他的各种操作请求进行控制，确保他只执行合法操作。

常用存取控制方法：

  • 自主存取控制：
    • 同一用户对于不同的数据对象有不同的存取权限
    • 不同的用户对同一对象也有不同的权限
    • 用户还可将其拥有的存取权限转授给其他用户
  • 强制存取控制：
    • 每一个数据对象被标以一定的密级
    • 每一个用户也被授予某一个级别的许可证
    • 对于任意一个对象，只有具有合法许可证的用户才可以存取

4.2.3 自主存取控制方法（P140）

自主存取控制（DAC）。

存取权限由两个要素组成：数据对象、操作类型。

在数据库系统中，定义存取权限称为授权。

优点：

  • 能够通过授权机制有效地控制其他用户对敏感数据的存取。

缺点：

  • 可能存在数据的“无意泄露”。原因：数据本身无安全性标记。解决：实施强制存取控制策略。

4.2.6 强制存取控制方法（P147）

强制存取控制（MAC）。

在MAC中，全部实体被分为主体和客体两大类：

  • 主体：系统中的活动实体
    • DBMS所管理的实际用户
    • 代表用户的各进程
  • 客体：系统中的被动实体，是受主体操纵的
    • 文件
    • 基表
    • 索引
    • 视图

敏感度标记：

  • 绝密（Top Secret）
  • 机密（Secret）
  • 可信（Confidential）
  • 公开（Public）

主体的敏感度标记称为许可证级别；客体的敏感度标记称为密级。
MAC机制就是通过对比主体和客体的敏感度，最终确定主体是否能够存取客体。

规则：

  • 仅当主体的许可证级别大于或等于客体的密级时，该主体才能读取相应的客体；
  • 仅当主体的许可证级别小于或等于客体的密级时，该主体才能写相应的客体。

第二条规则为了防止禁止了拥有高许可证级别的主体更新低密级的数据对象。例如，某TS级主体读取TS级数据后，将其写入P级数据中，造成了数据泄露。

MAC控制的特点：

  • MAC是对数据本身进行密级标记
  • 无论数据如何复制，标记与数据是一个不可分的整体
  • 只有符合密级标记要求的用户才可以操纵数据，从而提供了更高级别的安全性

数据库——数据库完整性

Posted on 2023-02-08

5.1 实体完整性（P158）

5.1.1 定义实体完整性（P158）

关系模型的实体完整性在 CREATE TABLE 中用 PRIMARY KEY 定义。

5.1.2 实体完整性检查和违约处理（P159）

当用户插入一条记录或对主码列进行更新时，数据库管理系统会进行检查：

  1. 检查主码值是否唯一，如果不唯一则拒绝插入或修改。
  2. 检查主码的各属性是否为空，只要有一个为空则拒绝插入或修改。

检查主码是否唯一的方式的一种方法是进行全表扫描，但该方法过于耗时，所以数据库管理系统一般会在主码上建立索引。

5.2 参照完整性（P160）

5.2.1 参照完整性（P160）

关系模型的参照完整性在 CREATE TABLE 中用 FOREIGN KEY 定义哪些列为外码，用 REFERENCES 指明这些外码参照那些表的主码。

5.2.2 参照完整性检查和违约处理（P160）

对参照表和被参照表进行删除、修改操作都可能破坏参照完整性。

当上述不一致发生时，系统可以采用如下策略加以处理：

  1. 拒绝执行（NO ACTION）（默认）。
  2. 级联操作（CASCADE）：当删除或修改被参照表的一个元组导致与参照表不一致时，删除或修改参照表所有导致不一致的元组。
  3. 设置为空值：当删除或修改被参照表的一个元组导致与参照表不一致时，将参照表所有导致不一致的元组的对应属性设置为空值。

外码能够取空值，要根据该外码在关系中是否同时作为主码来决定。所以对抑郁参照完整性，还应定义外码列是否允许空值。

5.3 用户定义完整性（P163）

5.3.1 属性上的约束条件（P163）

在使用 CREATE TABLE 定义属性时，可以同时对属性加以限制：

  • 列值非空（NOT NULL）
  • 列值唯一（UNIQUE）
  • 检查列值是否满足一个条件表达式（CHECK 短语）

向表中插入元组或修改属性的值时，如果不满约束条件，则直接拒绝执行。

5.3.2 元组上的约束条件（P164）

插入元组或修改属性的值时，关系数据库管理系统检查元组上的约束条件是否被满足，如果不满足则操作被拒绝执行。

5.4 完整性约束命名子句（P165）

SQL在 CREATE TABLE 语句中提供了完整性约束命名子句 CONSTRAINT。

格式：CONSTRAINT <完整性约束条件名> <完整性约束条件>

完整约束条件包括：NOT NULL、UNIQUE、PRIMARY KEY、FOREIGN KEY、CHECK。

可以使用 ALTER TABLE 修改表中的完整性约束限制。

5.6 断言（P167）

SQL中，可以使用 CREATE ASSERTION 语句，通过声明性断言来指定更具一般性的约束。

断言创建以后，任何对断言中所涉及的关系的操作都会触发关系数据库管理系统对断言的检查，任何使断言不为真值的操作都会被拒绝执行。

创建断言的语句格式：CREATE ASSERTION <断言名> <CHECK 子句>
每个断言都被赋予一个名字，<CHECK 子句>中的约束条件与WHERE子句的条件表达式类似。

删除断言的语句格式：DROP ASSERTION <断言名>

如果断言很复杂，则系统在检测和维护断言的开销较高，这是在使用断言时应该注意的。

5.7 触发器（P168）

触发器（Trigger）是用户定义在关系表上的一类由事件驱动的特殊过程。

触发器保存在数据库服务器中，任何用户对表的增、删、改操作均由服务器自动激活相应的触发器。
