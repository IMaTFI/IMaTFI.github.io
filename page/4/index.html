5.4 ARP协议

MAC地址

作用：在局域网中标识一个帧从哪个接口发出，到达哪个接口。

MAC地址通常为48位，固化在网卡ROM中，有时也可以用软件设置。e.g.：1A-2F-BB-76-09-AD（十六进制表示）

3.1 运输层概述

运输层的功能：为不同主机上的进程提供逻辑通信。

因特网上的运输层协议：

  • 用户数据报协议UDP（数据报）
  • 传输控制协议TCP（报文段）

3.2 端口与套接字

IP协议提供的复用/分用是将TCP报文段和UDP数据报分开。

3.3 无连接传输：UDP

UDP提供的功能：

  • 多路复用/分用
  • 差错检查

UDP的优势：

  • 无需建立连接：建立连接会增加时延
  • 简单：无需维护连接状态
  • 首部开销少：仅有8字节
  • 无拥塞控制：“想发就发”

UDP的问题：

  • 路由器中大量的分组溢出
  • 显著减小TCP通信速率，甚至挤垮TCP会话

3.4 数据可靠传输

数据在传输过程中不会丢失，在传输过程中不会出错。

3.4.1 可靠信道上的可靠传输

底层信道完全可靠，不会产生比特错误，不会丢失分组。

3.5 简单停等协议

场景：

  • 分组比特可能受损。
  • 所有传输的分组都会被按序接受，不会丢失。

初步考虑：

  • 判断分组受损——差错检测。
  • 通知发送方分组是否受损——接收方反馈（ACK、NAK）。
  • 得知分组受损后，发送方的处理手段——出错重传。

利用缓冲区缓存已发出但未收到反馈的报文段，接收方和发送方各一个报文段大小的缓冲区即可。

新的问题：ACK、NAK分组受损。
解决方案：将所有受损的分组视为NAK，并对分组进行编号从而避免重传。

实际的网络中，只使用ACK，ACK中必须指出被确认分组的序号

3.6 实用停等协议

场景：不仅有比特差错，还会丢包。

怎样检测丢包？ 超时重传。


至此，通过检验和、序号、计时器、确认分组等技术，我们实现了可靠的数据传输。
但是，停等协议传输效率较低。

流水线技术工作原理：

3.7.1 GBN协议

丢失的分组和其后的分组全部重传（重传发送方缓冲区内的所有分组），接收方只需缓存一个分组。

在GBN协议中，对一个分组进行确认，意味着其前的分组都被确认，称为累积确认。

在GBN协议中，只需设置一个计时器，负责记录序号为base的分组是否超时，若超时，则重传所有发送且还未确认的分组，并重启计时器；若没有超时，则重启计时器（若没有发送且还未确认的分组，则不需要重启计时器）。

3.7.2 SR协议

只重传丢失的分组，接收方的缓冲区需要和发送方一样大。

在SR协议中，需要为每个已发送且未确认的分组设置一个定时器。

3.8 TCP报文段首部结构

  • 源端口、目的端口：在运输层标识源进程和目的进程。
  • 序号：TCP报文段的数据部分的第一个字节在整个字节流中的序号。

  • ACK、确认号：当ACK为1时，确认号才有意义。确认号的含义为：期待收到的分组的序号。

    确认号可以实现捎带确认，即接收方在向发送方发送数据的同时，也确认了发送方之前发送的数据。

  • 首部长度：TCP报文段首部的行数（一行4B），最少5行，最多15行。

  • 窗口：用于流量控制。
  • URG、紧急指针：当URG为1时，将数据部分中紧急指针之前的所有数据全部紧急交付给上层。
  • PSH：PSH为1时，将全部数据紧急交付给上层。

3.11 TCP可靠传输机制

快速重传：如果发送方收到一个分组的3个冗余ACK，则认为该报文段丢失，重传该分组。

注意：对一个分组的第一个ACK不是冗余ACK。也就是说，如果发送方收到一个分组的4个ACK，则认为该报文段丢失。

超时间隔加倍：每次TCP重传均将下一次超时间隔设为先前值的两倍。

3.12 TCP的流量控制

如果发送方发送速度很快，而接收方的处理速度较慢，就会造成接收方的缓冲区被占满，导致大量的分组被丢弃，进而导致大量的分组被重传，产生了对信道资源的浪费。

接收方计算缓冲区中的空闲空间大小，并将其写入TCP报文的窗口字段中。发送方知道接收方缓冲区空闲空间的大小后，也将自己的发送缓冲区大小设置为该值。

考虑一种特殊情况：如果接收方的缓冲区满，且接收方对最后一个分组进行确认，则发送方会知悉接收方当前没有空闲区，于是发送方停止发送数据，接收方因而也不会产生任何反馈，通信进入一种“假死”状态。
所以在TCP的实际实现中，发送方知道接收方没有空闲区后，会持续发送数据大小为一个字节的数据。

3.13 TCP的连接管理

TCP连接的建立：

第三次握手会携带数据报文，所以可以认为TCP连接的建立需要一个RTT。

TCP连接的建立必须一次性完成，不允许分阶段完成。

TCP连接的释放：

TCP连接的释放可以分阶段完成。


如果服务器没有收到第三次握手，则会造成客户机进入了连接状态，而服务器没有进入连接状态。此时，服务器可以通过RST开关来尝试重新建立连接。

3.14 拥塞控制概述

拥塞：大量的分组堆积在网络核心，导致排队时延长，产生大量丢包。

端到端拥塞控制：

  • 网络层不为拥塞提供任何帮助。
  • 端系统通过对网络行为（丢包、实验增加）的观测判断网络是否发生拥塞。

6.1 问题的提出（P178）

针对一个具体的问题，应该构造几个关系模式，每个关系有哪几个属性组成。

第一范式：每一个分量必须是不可分的数据项。

数据依赖是一个关系内部属性之间的一种约束关系，主要有函数依赖和多值依赖。

函数依赖类似于数学中的函数 ，自变量 确定后，函数值 也就唯一确定了。如：学生学号Sno确定后，学生姓名Sname也确定了，即Sno函数决定Sname，或Sname函数依赖于Sno，记作SnoSname。

一个不好的关系模式可能产生如下问题（以上图为例）：

  1. 数据冗余
    如：每个系的系主任名重复出现。
  2. 更新异常
    由于数据冗余，当更新数据库中的数据时，系统往往要付出很大的代价来维护数据库的完整性。如：某系更换系主任后，必须修改与该系学生所有有关的元组。
  3. 插入异常
    如果一个系刚刚成立，尚无学生，则无法把系和系主任的信息存入数据库。
  4. 删除异常
    如果某个系的学生全部毕业了，这个系和系主任的信息也被一并删除了。

一个好的关系模式不应发生更新异常、插入异常、删除异常，同时数据冗余应该尽可能少。

实体与属性的划分原则

为了简化E-R图，现实世界的事务能作为属性，就作为属性。

符合下述两个条件的事物，可以作为属性：

  1. 属性必须是不可分的数据项，不能包含其他属性。
  2. 属性不能与其他实体具有联系。

E-R图的集成

在开发一个大型信息系统时，常常要首先设计各子系统的分E-R图，然后将它们集成起来。E-R图的集成一般分为以下两步：

  1. 合并，解决各E-R图之间的冲突。
  2. 修改和重构，消除不必要的冗余。

各子系统的E-R图之间的冲突主要有3类：

  1. 属性冲突：
    • 属性域冲突： 属性值的类型、取值范围、取值集合不同。
    • 属性取值单位冲突。
  2. 命名冲突：
    • 有同名异义、异名同义两种。
    • 命名冲突可能发生在实体、联系、属性上。

  3. 结构冲突

    • 同一对象在不同应用中有不同的抽象。

      如：“职工”在某些局部应用中被当作实体，在另一局部应用中被当作属性。

    • 同一主体在不同子系统的E-R图所包含的属性个数和属性排列次序不完全相同。

    • 实体间的联系在不同E-R图中为不同的类型。

消除冗余主要通过分析法。

7.4 逻辑结构设计（P231）

7.4.1 E-R图向关系模型的转换（P231）

一个实体型转换为一个关系模式，关系的属性就是实体的属性，关系的码就是实体的码。

对于实体间的联系，有如下不同的情况：

  1. 一个1：1联系可以转换为一个独立的关系模式，也可以与任意一端合并。如果转换为一个独立的关系模式，则各实体的码以及联系本身的属性均转换为关系的属性；如果与某一端实体对应的关系模式合并，则需要在该关系的属性中加入另一个关系模式的码和联系本身的属性。
  2. 一个1：n联系可以转换为一个独立的关系模式，也可以与n端的关系模式合并。如果转换为一个独立的关系模式，则与该联系相连的各实体的码以及联系本身的属性均转换为关系的属性，而关系的码为n端实体的码。
  3. 一个m：n联系转换为一个关系模式。与该多元联系相连的各实体的码以及联系本身的属性均转换为关系的属性，各实体的码的组成关系的码。
  4. 具有相同码的关系模式可合并。

操作系统——处理机调度

Posted on 2022-11-13

1. 调度的三个层次

1.1 作业调度

作业调度决定是否为作业创建相应的进程（建立PCB），并将其加入将要执行的进程集合中。

作业调度是外存与内存之间的调度。每个作业只会被调入一次、调出一次，作业调度主要关注调入问题，因为只有调入的时机需要操作系统决定。

1.2 内存调度

引入虚存技术后，可以将暂时不能运行的进程移至外存等待，即变为挂起状态。处于挂起状态的进程的PCB会常驻内存，被放入挂起队列中。

内存调度调度就是决定是否将一个挂起进程载入内存。

一个进程可能被多次调入、调出内存，因此内存调度的发生频率比作业调度高。

1.3 进程调度

进程调度决定哪个就绪进程将被处理机执行。

进程调度的频率很高，一般几十毫秒一次。

1.4 三层调度的对比

2. 进程调度

2.1 进程调度的时机

需要进行进程调度、切换的情况：

  • 当前进程主动放弃处理机：
    • 进程正常终止
    • 运行过程中发生异常而终止
    • 进程主动请求阻塞（如I/O请求）
  • 当前进程被动放弃处理机：
    • 时间片用完
    • 有更紧急的事需要处理（I/O中断）
    • 有更高优先级的进程进入就绪队列

不能进行进程调度、切换的情况：

  • 在处理中断的过程中
  • 进程在操作系统内核程序临界区
  • 在原子操作过程中

2.2 进程调度的方式

被抢占方式：只允许进程主动放弃处理机。这种方式实现简单，但无法处理紧急任务，适合早期的批处理系统。

抢占方式：允许进程被动放弃处理机。

2.3 进程的切换与过程

进程切换的过程主要完成了：

  1. 对原来运行进程的各种数据的保存。
  2. 对将要运行进程的各种数据的恢复。

    各种数据包括：程序计数器、程序状态字、数据寄存器等，这些信息一般保存在进程控制块中。

由此可见，进程切换是有代价的，过于频繁的切换可能会导致系统效率的降低。

3 调度算法的评价指标

3.1 CPU利用率

CPU“忙碌”的时间占总时间的比例。

3.2 系统吞吐量

单位时间内完成作业的数量。

3.3 周转时间

从作业被提交给系统开始，到作业完成为止的这段时间间隔。它包括4个部分：

  1. 作业在外存上等待作业调度的时间。
  2. 进程在就绪队列上等待调度的时间。
  3. 进程等待I/O操作完成的时间。

作业周转时间 = 作业完成时间 - 作业提交时间

平均周转时间 = 各作业周转时间之和 / 作业数

带权周转时间 = 作业周转时间 / 作业实际运行的时间

平均带权周转时间 = 各带权周转时间 / 作业数

3.4 等待时间

指进程/作业处于等待处理机状态的时间之和。

对进程来说，等待时间是指进程建立后，等待被服务的时间之和。注意：在等待I/O完成的期间，其实也是在被I/O设备服务的，所以不计入等待时间。

对作业来说，相比进程还要额外考虑作业在外存后备队列中的等待时间。

等待时间 = 周转时间 - 运行时间 - I/O操作时间

一般而言，一个作业/进程需要CPU、I/O设备服务多久都是固定的，因此调度算法只会影响作业/进程的等待时间。

3.5 响应时间

用户提交请求到首次产生响应的时间。

响应时间 = 进程第一次被CPU执行的时刻 - 提交时间（不确定）

4 调度算法

4.1 先来先服务

从“公平”的角度考虑，按照作业/进程到达的先后顺序进行服务。

从另一个角度说，优先为等待时间最长的作业/进程服务。

对于作业调度，考虑的是哪个作业先到达后备队列；对于进程调度，考虑的是哪个进程先到达就绪队列。

非抢占式算法。

优点：公平、实现简单，不会导致饥饿。
缺点：对长作业不利、对短作业不利。

4.2 短作业优先

为了追求最少的平均等待时间、最少的平均周转时间、最少的平均带权周转时间，让服务时间最短的作业优先得到服务。

短作业优先算法默认是非抢占式的，每次调度时选择当前已到达且运行时间最短的作业。

抢占式的短作业优先算法又称“最短剩余时间优先算法”，每当有进程加入就绪队列或进程完成时就需要进行调度，选择剩余执行时间最短的作业。

优点：“最短的”平均等待时间、平均周转时间
缺点：不公平；对短作业有利，对长作业不利；可能导致饥饿，甚至“饿死”；作业的运行时间由用户提供，不一定真实。

4.3 高响应比优先

综合考虑作业/进程的等待时间和要求服务的时间，在每次调度时选择响应比最高的作业/进程为其服务。

响应比 = （等待时间+要求服务时间）/ 要求服务时间

非抢占式算法。

优点：综合考虑了等待时间和运行时间，不会导致饥饿。

4.4 时间片轮转调度算法

按照各进程到达就绪队列的顺序，轮流让各个进程执行一个时间片（如100ms）。若进程未在一个时间片内执行完，则剥夺处理机，将该进程放到就绪队列的队尾重新排队。

如果一个进程时间片到的时刻，一个新进程恰好进入就绪队列，则默认新进程排在前面。

该算法适用于进程调度，属于抢占式算法，由时钟装置发出时钟中断来通知CPU时间片到。

若时间片过长，则时间片轮转调度算法退化为先来先服务算法，大大增加进程响应时间；若时间片过短，则会导致进程调度过于频繁，进程切换的开销过大。

优点：公平；响应快；不会导致饥饿。
缺点：不区分任务的紧急程度。

4.5 优先级调度算法

调度时选择优先级最高的作业/进程。

非抢占式、抢占式均有。非抢占式的进程调度只发生在进程主动放弃处理机时，而抢占式的进程调度还会发生在就绪队列变化时。

优先级有静态优先级和动态优先级两种：

  • 静态优先级：进程创建时决定，之后一直不变。
  • 动态优先级：进程创建时有一个初始值，之后根据情况动态调整优先级。

通常：

  • 系统进程优先级高于用户进程
  • 前台进程高于后台进程
  • I/O型进程高于计算型进程

    解释：如果让I/O型进程优先运行，则I/O设备可以尽早投入使用，以实现I/O设备和CPU的并行工作，提高资源利用率、系统吞吐量。

处于公平、资源利用率等角度考虑：

  • 如果某进程在就绪队列中等待时间过长，则适当提升其优先级。
  • 如果某进程占用处理机时间过长，则适当降低其优先级。

优点：可以区分进程的重要程度、紧急程度；灵活。
缺点：可能导致饥饿。

4.6 多级反馈队列调度算法

设置多级就绪队列，各级队列优先级从高到低，时间片从小到大。新进程到达时，先进入第1级队列，按照先来先服务原则排队等待时间片，若时间片到进程还未结束，则进程进入下一级队列队尾，如果此时已经在最下级队列，则重新放回该队列队尾。只有前k级队列为空时，才会为第k+1级队头的进程分配时间片。

抢占式算法。在k级队列的进程运行过程中，若上级队列进入了一个新进程，则此新进程会抢占处理机，原来运行的进程放回k级队列的队尾。

优点：综合上述算法的特点。
缺点：可能导致饥饿。

5.3 多路访问协议

5.3.1多路访问控制（MAC）协议

两类链路：点对点链路、广播链路（共享介质）。

设计目标

采用分布式算法，每个结点可以自己做出决策，决定何时可以传输数据。
基于信道本身进行协调，无需额外的协调通道。

理想MAC协议

给定：速率为 bps的广播信道。
期望：

  • 有个结点同时发送数据是，每个结点平均发送速率为。
  • 完全分散控制：无需特定结点协调；无需时钟、时隙同步。
  • 实现简单。

MAC协议分类

  1. 信道划分MAC协议：采用多路复用技术。
  2. 随机访问MAC协议：信道不划分、允许冲突；采用冲突恢复机制。
  3. 轮转MAC协议：结点轮流使用信道。

时隙ALOHA协议


优点：

  • 单个结点活动时，可以以信道全部速率进行传输。
  • 高度分散化：只需时钟同步。

  • 发生冲突时，会浪费时隙。

四边形不等式优化

对状态转移方程：

注意：一定是min！

如果二元函数 $w(i,j)$ 满足下述条件：

  1. 区间单调性：对 $l \leq l’ \leq r’ \leq r$ ，有 $w(l’,r’) \leq w(l,r)$ 。即“被包含者不大于包含者”。
  2. 四边形不等式：对 $l \leq l’ \leq r’ \leq r$ ，有 $w(l,r’)+w(l’,r) \leq w(l,r)+w(l’,r’)$ 。即“交叉不大于包含”

可以得到如下结论：

  1. 当 $w(l,r)$ 满足上述条件时， $dp[i][j]$ 也符合四边形不等式。
  2. 如果$dp[i][j]$满足四边形不等式，设 $m[i][j]$ 为 $dp[i][j]$ 对应的最优决策点，则有 $m[l][r-1] \leq m[l][r] \leq m[l+1][r]$。


此时，我们在动态规划时， $k$ 的枚举范围就从 $[l,r]$ 变成了 $[m[l][r-1],m[l+1][r]]$ 。

区间DP的DP顺序是从主对角线开始，一条条对角线地往上转移的。在上图中，对于每一条对角线，考察它枚举的区间，可以发现每两个相邻的区间只有一个元素重叠，所以处理每条对角线需要花费 $O(n)$ 的时间，处理所有对角线也就只需要 $O(n^2)$ 的时间。

思路

这道题目和例1有些类似。根据题目要求，每个数字只能与相邻的数字合并，所以我们尝试使用区间dp求解。

定义状态dp[i][j]为区间中的数字完全合并后得到的数字。所谓完全合并，是指区间内的数字经过一系列合并后，最终合成为一个数字。对于不能完全合并的区间，我们不需要对其进行处理。

根据题目要求，只有当dp[i][k]和dp[k+1][j]的值相等时，两个区间才能够合并，此时有dp[i][j] = max(dp[i][j], dp[i][k]+1)。于是，我们只需要求得每个区间完全合并后的值，并在求值的过程中记录最大值即可。
